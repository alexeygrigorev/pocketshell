# Decisions Log

Captured 2026-05-21 during conceptual planning, before any code.

## Locked

| # | Decision | Rationale |
|---|---|---|
| D1 | App name PocketShell, package `com.pocketshell.app` | Matches working directory; clean, available-looking. |
| D2 | Composition, not fork | Extract reusable parts of `ssh-auto-forward-android` into shared Gradle modules. Both apps consume them. Neither reaches into the other. |
| D3 | sshj, not JSch | JSch is unmaintained; sshj supports ed25519 and modern KEX. Do the swap during extraction, not after. |
| D4 | Vendored Termux `terminal-emulator` + `terminal-view` | Battle-tested xterm-256color. Writing a VT emulator from scratch is a 6-month detour. |
| D5 | `tmux -CC` control mode, not screen-scraping | Structured protocol gives us per-pane state. The single most important call for tmux-native UX. |
| D6 | Per-pane rendering with swipe navigation, not tiled tmux | Tmux's split layout is unreadable on a phone. PocketShell renders one pane at a time; PocketShell *is* the status bar. |
| D7 | Server-side recurring jobs via `pocketshell jobs daemon` on each host | Jobs survive phone offline / sleep. Trade-off: host bootstrap is a first-class onboarding step for the unified `pocketshell` helper. |
| D8 | Termius-inspired design language, built in shared `ui-kit` module | Both apps converge on one visual language. Avoids per-screen redesign. |
| D9 | Voice input is first-class. Engine: Whisper via OpenAI Audio Transcriptions API. | Better than Android `SpeechRecognizer` for technical content. Per-request cost accepted. Existing `openai-transcribe` skill is the integration reference. |
| D10 | Mic trigger: tap-to-toggle, auto-stop after configurable silence (30s default, 2s-60s range) | Best for long agent prompts. Hold-to-talk is fatiguing past 10s. The conservative default avoids cutting off natural pauses while the setting still allows more aggressive auto-stop when wanted. |
| D11 | Prompt composer is a bottom sheet (modal over terminal) | Cleanest hierarchy for review-and-send. Terminal dims behind. |
| D12 | Key bar always visible above the keyboard (only while keyboard is up) | One-tap Esc/Tab/Ctrl/Alt/arrows. Matches "super smooth" requirement, zero discovery cost. |
| ~~D13~~ | ~~Chord palette opens on long-press of ⚡ in the key bar~~ | Superseded by D18. |
| D14 | Agent-aware conversation view — read THIS session's JSONL directly over SSH | Solves the "can't scroll back through agent output on mobile" problem. Scope is current pane / current session only — not historical search (that's agent-log-explorer's job). No host dependency. |
| D15 | Conversation surface: tab at top of session view `[Terminal] [Conversation]` | Tab hidden when no agent detected. The conversation surface supports reply-in-place to the current agent pane. Hint chip in terminal view appears once per detected session for discoverability. |
| D16 | Agent detection: tmux pane cwd + recent log activity + pane-scoped process confirmation | Runtime-enabled for Claude Code (`~/.claude/projects/<encoded-cwd>/`), Codex (`~/.codex/sessions/` filtered by rollout cwd), and OpenCode (`~/.local/share/opencode/opencode.db` filtered by session directory / project worktree). Tmux pane callers require a matching process on the pane's TTY before showing the Conversation tab. Silent fallback when nothing matches. |
| D17 | New shared module `core-agents` for parsers + detection | Unit-testable without SSH. Each agent has its own parser producing a normalized `ConversationEvent` stream consumed by the UI. |
| D18 | Drop chord palette and ⚡ lightning key from v1. Key bar shrinks to 8 slots (Esc, Tab, Ctrl, Alt, ←↑↓→) | User feedback: detach + session switch are the only chords they actually want, and both are better served by native UI (back arrow = detach, tap session name = switcher). Power-user chord palette may return as opt-in post-v1 if demand appears. |
| D19 | Usage panel data is fetched server-side over SSH. Default source: `pocketshell usage --json`. Pluggable per host (user can override command). | PocketShell holds zero provider credentials (no OAuth, no API keys for Anthropic/OpenAI/GitHub). If a host doesn't have the helper installed, that host is omitted from the usage panel until setup passes. |
| D20 | New shared module `core-usage` to parse normalized JSON from server-side usage tools | Pluggable parsers per provider. Same compose-the-server-tool pattern as `core-agents`. |
| D21 | PocketShell does not run in the background, with foreground-service carve-outs for auto-forward and live sessions plus a bounded grace window for terminal ride-through (locked 2026-05-27, issue #161; auto-forward carve-out 2026-05-27, issue #203; grace window 2026-06-03, issue #450; session carve-out 2026-06-27, issue #977) | App must not consume battery while the user is not interacting with it except for explicit user-visible foreground-service holds. No `WorkManager`, no `AlarmManager`, no `setRepeating`, no `Timer`, no `ScheduledExecutorService`. Periodic schedulers (today: `UsageScheduler`) hook `ProcessLifecycleOwner` and pause on `Lifecycle.State.STOPPED`; they resume on `STARTED`. Tmux on the remote keeps the long-running state; the client reconnects on next foreground unless a sanctioned foreground-service hold is active. The Active Sessions home-screen widget is allowed (system-driven, not the app holding wakelocks). **Carve-out (issue #203 expanded scope):** the auto-forward feature MAY run a foreground service while ≥1 tunnel is active, because the tunnels are bound to the device-side SSH transport and die the moment the app process backgrounds — without a service, the user loses the entire feature's value (port forwards to `localhost:N` go dark). The carve-out is scoped: ONLY the auto-forward path, ONLY while the user has at least one active or auto-forward-enabled host, with a visible persistent notification, and torn down the moment the user disables auto-forward across all hosts. **Carve-out (issue #977):** a live attached SSH/tmux session MAY run a foreground service with a `PARTIAL_WAKE_LOCK` while ≥1 tmux client is connected. The service has a visible persistent notification, starts from the live-client registry, and stops when the last live client disconnects/unregisters. While this hold is active, the destructive background-grace teardown (tmux detach + SSH lease stop) is suppressed so the user's main interactive session remains connected through background/Doze. **Bounded grace window (issue #450, maintainer-sanctioned relaxation):** the terminal SSH/tmux teardown that previously fired immediately on `ON_STOP` is now *delayed* by a single bounded window (`App.BACKGROUND_GRACE_MILLIS`, 60 s) so a quick app-switch — copying a snippet, glancing at another app — returns to PocketShell before teardown runs and resumes the live connection instantly with no reconnect and no "Connecting"/"Reconnecting" UI. This is a one-shot, self-cancelling `delay` of an already-scheduled teardown, cancelled on the next `ON_START`; if no foreground-service hold is active when it elapses, staying backgrounded past the window runs the full teardown exactly as before. The usage poll, agent detection, and voice composer keep pausing immediately on `ON_STOP` — the grace window / session hold applies ONLY to the active terminal connection. |
| D22 | No backwards-compatibility: hard cuts only (locked 2026-05-27; Room update carve-out 2026-06-01, issue #386) | The maintainer is the only user of PocketShell today. There is no third-party API to honor and no SDK consumer to warn. Every implementer and reviewer must assume hard-cut semantics for feature replacement: when a feature replaces an older one, delete the older in the same PR — no legacy detection path, no deprecation shim, no settings flag for "use the old behaviour", no fallback branch. Room-backed user data is now the explicit carve-out: normal same-package APK updates must preserve hosts, key metadata, roots, sessions, snippets, cost history, pending transcription metadata, and port-forwarding config. When a Room schema changes, ship a migration and do not use production `fallbackToDestructiveMigration`, downgrade fallback, or startup open-failure database deletion as the routine update path. A destructive reset is only acceptable as an explicit user-confirmed recovery/reset path or as a separately reviewed release decision with a preservation/export plan. When porting from upstream (e.g. `ssh-auto-forward-android`, the unified `pocketshell` daemon, etc.), DO NOT port legacy-detection shims that the upstream kept for its own install-base reasons. The orchestrator removes legacy code proactively as part of every round, not as a separate cleanup task. |
| D23 | GitHub-aware navigation goes through the dev box's `gh` CLI, not phone-side OAuth (locked 2026-05-27, issue #205) | Mirrors D19's "zero provider credentials on the phone" stance. `pocketshell repos list --remote` and any future clone-on-tap operation run server-side via the maintainer's existing `gh auth` setup on the dev box; the phone is a viewer. Three forcing reasons: (1) precedent — `UsageRemoteSource.kt` already says "never asks for provider OAuth/API keys"; (2) the clone target IS the dev box, so listing on the phone but cloning on the dev box is a split-actor dead-end; (3) zero new auth surface on the phone. The `gh` token scope can be narrowed to `repo:read` for browse-only flows; `gh repo clone` uses SSH keys (no PAT write scope). Audit token scope as part of `HostBootstrapper`. |
| D24 | Env files (.env/.envrc) edited via server-side `pocketshell env` (locked 2026-05-29, issue #262) | Values written via stdin JSON never argv; both files managed when present (each key tagged with its file); new keys default to `.env`; plain reveal, no biometric. |
| D25 | In-app action assistant uses a provider-agnostic `AssistantLlmClient` with a unified tool-call shape (locked 2026-05-29, issue #265) | The in-app action assistant talks to LLMs through a provider-independent `AssistantLlmClient` (`complete(messages, tools, toolChoice)` returning a unified `LlmResponse` of text + tool calls + stop reason). Product-facing providers are OpenAI, Anthropic, and ZAI, each with its own base URL, model, and KeyStore-backed secret slot in settings. ZAI reuses the Anthropic-compatible Messages wire implementation internally (`https://api.z.ai/api/anthropic/v1`, `glm-*` models; the older root `https://api.z.ai/api/anthropic` is normalized to `/v1/messages`), but it is not configured as Anthropic. The OpenAI chat-completions impl is separate. Default provider is OpenAI. Voice-note transcription stays on Whisper/OpenAI independently — its key and code path are untouched. |
| D26 | Agent stop/idle detection via `pocketshell hooks` installing per-engine hooks (Claude `Stop`/`SubagentStop`/`Notification`, Codex `notify`, OpenCode plugin); install MERGES into existing settings (never clobbers), uninstall removes only our entries; events normalized to a JSONL bus; integration only. (locked 2026-05-29, issue #267) | The empirically-verified `stop-handle` PoC proved one cross-engine protocol: each engine emits a stop/idle/waiting-for-input signal that normalizes to `{ts, engine, state, source, session_id, cwd, ...}` on an append-only JSONL bus. PocketShell ships this server-side via `pocketshell hooks install/uninstall/status/events`. Per D22 (hard cuts) there is no legacy-detection shim, but install is still non-destructive on the *user's* config: Claude hooks merge into `~/.claude/settings.json` preserving other keys + pre-existing hooks; Codex sets the top-level `notify` in `~/.codex/config.toml` only when unset (warn+skip if pointed elsewhere); OpenCode drops a plugin file without disturbing other plugins. `uninstall` removes only our entries and is idempotent. Handlers + bus live under `~/.cache/pocketshell/hooks/`. Integration only — the "tell the agent to continue" supervisor UX is deferred. |
| D27 | Canonical agent-trace + app/crash log sink via server-side `pocketshell logs`; Tier 1 = single canonical host (the dev box), no cloud. (locked 2026-05-29, issue #270) | The assistant's actions are, by design (D19/D23), all server-side `pocketshell` calls over SSH, so there is already a server-side choke point on every meaningful event; volume is tiny (KB/day, append-only JSONL); the dev box is already the hub. Putting the canonical sink on the dev box makes the record persistent + survives app deletion, readable even if the phone app crashes on startup (the orchestrator can `rg` the JSONL directly with no SDK — the key requirement), and adds zero new credential surface (the phone holds no cloud creds, D19). `pocketshell logs ingest` reads one JSON event from stdin, aggressively redacts secrets (secret-named keys, token-shaped strings, inline `KEY=value` secret assignments — deny-by-default), stamps `ts`/`schema`, and appends to `${XDG_STATE_HOME:-~/.local/state}/pocketshell/logs/{agent,app}-YYYYMMDD.jsonl` (0600). `tail`/`path` read/locate the record; `import-hooks` mirrors the #267 hooks bus in as `kind=engine_event` idempotently (byte cursor, no duplicates). Every event carries `target_host` + `install_id`. S3/cloud was considered and rejected for Tier 1 (it adds AWS credential brokering for ~no benefit at this volume); off-box durability is deferred to a later Tier-2 issue, with a private git mirror preferred over S3. |
| D28 | The SSH/tmux Connection Manager is the single most critical subsystem and is managed as a first-class architecture, never via patches-on-patches (locked 2026-06-13, maintainer directive) | The connection / lease / reconnect / grace core is the #1 dogfood blocker and the behaviour the maintainer feels most. Rules: **(1) Rewrite over patch.** When the current architecture stops allowing clean extension, prefer a clean-slate rewrite over stacking another compatibility branch / shim / special-case onto the reconnect path. This is the epic #687 clean-slate mandate, hard-cut per D22. **(2) Flag cardinal rework — don't silently keep patching.** The orchestrator continuously watches the connection core's architectural health; the moment it judges the design can no longer be extended cleanly and needs cardinal rework, it STOPS and tells the maintainer explicitly. The maintainer owns the rewrite-vs-patch call. **(3) Tests are preserved + expanded** across any rewrite; the load-bearing journeys (background→foreground within the D21 grace window, multi-session switch, reconnect/EOF) must run in per-PR CI (#638/#691), and a journey test must fail when the *user-visible* behaviour regresses — not merely when an internal/shadow state diverges. **(4) Finishing #687 = single active path.** "Done" means the new `core-connection` ConnectionController is the SOLE active path for connect/attach/reattach/grace/lease, and the old `TmuxSessionViewModel` reconnect/grace path is DELETED — no shadow/old coexistence remains. The current half-migrated shadow+old state is itself the patches-on-patches condition this decision exists to end. |
| D29 | The `ConnectionController` turn-on is phased and gated by device-truth journeys, with a TEMPORARY user-facing "Connection: New/Old" Settings toggle (default New) as the on-device fallback, removed together with the old path in #766 (locked 2026-06-14, epic #687) | The new `core-connection` `ConnectionController` (the D28 single-active-path target) is being turned ON behind a user-facing New/Old toggle, **phase by phase** — each phase proven on-device before the next reveals — rather than as one big-bang cutover. **Phases:** **P1 — screen-keyed reveal (#686, MERGED):** the screen state machine (`Navigating(targetId) → Seeding(targetId) → Live(targetId)`) keys every pane seed to its target session id and drops any seed whose id ≠ the current target, so a late seed from a superseded switch can never paint the wrong session. **P2 — single grace owner (#635, MERGED):** the within-grace foreground probe / ride-through is owned by one grace machine anchored on the lease's 60 s warm window (D21), collapsing the previously-disagreeing grace clocks that caused #685. **P3 — reseed-on-reattach (#553, PENDING):** reattach over a stale lease reseeds panes (no blank pane, no surfaced `list-panes` EOF). **P4 — Codex backpressure (#576, PENDING):** flow-control / backpressure for high-volume agent (Codex) output. **Gating — device-truth journeys J1–J4:** each phase is gated by a journey that asserts the **user's rendered viewport** (the terminal text the user actually sees), NOT internal/shadow state. This is the direct lesson from the green-CI-shipped-broken-journeys failure (#636/#638): a journey that only checks internal state can be green while the visible app is broken, so the gate must fail when the *user-visible* output regresses (the D28(3) rule). **Temporary toggle (sanctioned D22 exception):** while the turn-on is in flight, a user-facing **"Connection: New/Old"** Settings toggle (default **New**) lets the maintainer fall back on-device to the old inline `TmuxSessionViewModel` path if a phase misbehaves in real use. This is an explicit, time-boxed exception to D22's "no settings flag for old behaviour" — sanctioned only because it is the on-device safety valve for the most critical subsystem (D28) during its turn-on. The toggle and the entire old path are DELETED together in **#766** once the maintainer confirms the new path on-device. After #766 there is a single active path per D28(4): no toggle, no old `TmuxSessionViewModel` reconnect/grace path. Cross-refs: **D22** (hard-cut — this toggle is the one sanctioned, temporary exception, removed in #766), **D28** (connection manager is the most critical subsystem, managed first-class). |
| D30 | PocketShell does not manage tmux WINDOWS; windows created outside PocketShell are surfaced as separate `[wN]` switcher entries (hard-cut, locked 2026-06-15, issue #782) | The in-session window-tab row (`Window 1 ×` / `Window 2` / `+ window`) competed with the Terminal/Conversation tabs and confused the session model, and PocketShell-created windows made a session harder to navigate *outside* the app. So PocketShell **no longer creates, switches, renames, or closes tmux windows** — all in-app window-management UI and backing commands are DELETED per D22 (no flag, no legacy path): the `WindowStrip` row, the `WindowSwitcherOverlay`, the `+ New window` / `Switch window` / `Rename window` / `Stop window` kebab items, and the `TmuxSessionViewModel.newWindow`/`selectWindow`/`renameWindow`/`killWindow` + `windowKillError`/`windowSwitchRequest` machinery. Windows that *already exist* on the server (created outside PocketShell) are re-presented: when a tmux session has >1 window, the host tree lists **each window as its own session-like switcher entry** suffixed `[wN]` (e.g. `mysession [w0]`, `mysession [w1]`); tapping one attaches to THAT window's pane via the existing `initialWindowIndex` → unified-pager scroll, which reuses the warm lease (instant-switch, D21 / #636 / #687 — no reconnect). A single-window session is unchanged (no `[wN]` suffix). With the window-tab row gone, **Terminal/Conversation is the only in-session tab dimension**. Cross-ref: **D21** (warm-lease instant-switch the `[wN]` entries honour), **D22** (hard-cut). |
| D31 | Durable-fix gate: the reviewer defaults to `CHANGES REQUESTED`, and a reopened/recurring issue MUST ship with a class-covering, gate-wired regression test or it is rejected (locked 2026-06-19, maintainer directive) | The maintainer's #1 process complaint is that issues get closed as fixed, the reviewer approves, we ship, and the symptom returns — repeatedly (142 issues carried recurrence/regression language by 2026-06-19). Reopening "fixed" issues is the biggest waste in the project, and the cause is a reviewer that verifies the single reported instance, resolves uncertainty in the implementer's favour, and waves changes through "because the area is in active rework". Rules: **(1) Default reject.** The reviewer's default verdict is `CHANGES REQUESTED`; `APPROVED` must be earned criterion-by-criterion from artifacts the reviewer produced this run. Unproven ≠ passing; uncertainty ⇒ reject. "Active rework" is never grounds to approve a broken/unproven change — the reviewer turns it back and says so explicitly. **(2) Durable test mandatory for reopens.** A reopened/recurring issue (reopened event, "recurrence/still/again/regress" language, or a sibling that closed the same symptom) must add an automated regression test that (a) fails on the bug (red→green proven), (b) covers the whole CLASS — other sites/sessions/agent-kinds/states/the missing-data case — not just the one reported instance, (c) reproduces the maintainer's exact scenario (F2/F3), and (d) runs in per-push CI or the pre-tag gate. No durable test ⇒ reject; not waivable. **(3) Active-rework adjacency sweep.** For the connection/reconnect/lease core, the session tree, conversation-source/agent detection, or the composer/bottom chrome, the reviewer confirms the change does not reintroduce a recently-CLOSED sibling symptom and runs that area's load-bearing journey; a resurrected sibling symptom is blocking even when the issue's own ACs pass. **(4) Orchestrator flags reopens** in the reviewer brief so the gate is applied, and must not merge a reopened-issue fix lacking a durable regression test. Mechanics live in `.claude/agents/reviewer.md`. Cross-refs: **D28** (don't let the critical path regress), **#657/#638** (regression-proof test validity, journeys in per-push CI). |
| D33 | Definition of Done = "verified gone", not "change landed": reproduce → fix → verify → report, binding implementer, reviewer, and orchestrator (locked 2026-06-22, maintainer directive) | The maintainer's standing complaint is that things are reported fixed and aren't. The cause is treating "the change is committed / a test is green somewhere" as done. D33 makes the bar explicit and three-way. **(1) Implementer reproduces FIRST:** land a test that reproduces the exact reported scenario on the real path (on-device/connected journey for on-device reports; the production screen state, not a proxy), see it fail **red**, add the **fixture** that creates the non-happy state if needed (a happy fixture proves nothing — the #847 lesson), then fix to **green**, re-run, and only then report with red→green artifacts. **(2) Reviewer independently RUNs it:** sees the test fail red on base and pass green with the fix *this run*, and for user-facing flows reproduces the actual journey on emulator+Docker and confirms the user-visible symptom is *actually gone* from authoritative artifacts — not that an assertion passed. The review comment states "reproduced red on base, confirmed gone green, here is the artifact". Default `CHANGES REQUESTED`; unproven/uncertain ⇒ reject; correct-but-unprovable-on-real-path ⇒ `BLOCKED`, never `APPROVED`. **(3) Orchestrator never reports "ready" without the proof in hand:** before telling the maintainer a fix is done, confirm the red→green + symptom-gone artifact exists and the gate passed; otherwise say "not yet verified" plainly. Reporting a not-verified fix as done is itself a violation. Environment can't enter the state ⇒ inject it synthetically and hard-fail (the #780 model), never self-skip the load-bearing assertion. This is the one-line contract that D31/D32 and gates G1–G10 implement. Cross-refs: **D31/D32** (durable-fix + universal approval gates), **#657/#638/#847** (test validity, reproduce-first end-to-end). |
