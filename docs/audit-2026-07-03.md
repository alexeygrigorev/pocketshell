# Codebase stability + UX audit — 2026-07-03

Read-only audit run by five parallel auditors (connection/SSH core, terminal/tmux
render pipeline, app lifecycle/storage, agent-awareness/voice/usage, and a
UX jobs-to-be-done pass). Every finding below was traced in the actual code;
findings that duplicate already-open issues (#1219, #1214, #1208, #1164, #1158,
#1155, #1123, #1073, #1047, #928, #895, #874, #859, #843, etc.) were excluded.
GitHub issues filed from this report are listed in the issue map at the bottom.

Overall verdict: the codebase is heavily hardened and shows the scars of the
D31/D32/D33 regime — most obvious defect classes are already fixed. The
remaining risk clusters in four places:

1. **Lease/transport liveness disagreement** introduced by the async-`close()`
   change (#1144) — the one confirmed high-severity connection defect.
2. **All-or-nothing parsing at trust boundaries** (usage panel, agent-log
   envelope, Codex detection) — the #847 version-skew class is still open in
   three places.
3. **Latent cliffs on rare paths** (corrupt prefs crash-loop, multi-chunk QR
   share crash, never-closed `%begin` block).
4. **UX: the multi-host/multi-agent supervision job** — the core product
   thesis — is the weakest-served job in the shipped app.

## Part 1 — Stability findings

### 1.1 Connection / SSH core

The `ConnectionController` reducer + effect-driver split is clean, deterministic
and well-tested; the core is **not** at the D28 patches-on-patches threshold.
The strain concentrates in the lease/transport liveness layer.

#### [HIGH] C1 — Keepalive-dead transport retained as a warm-idle lease (stale `isConnected` during async close)

- **Where**: `shared/core-ssh/.../SshLeaseManager.kt:587-615` (`release()` keep-warm
  decision); root in `RealSshSession.kt:1734-1904` (async `close()`, #1144) and
  `TransportDispatcher.kt:210-214`; consumer at
  `app/.../TmuxSessionViewModel.kt` (`releaseCurrentLeaseOrCloseRawSession`).
- **What**: After #1144, `close()` returns immediately; `isConnected` stays
  `true` for up to ~2 s (longer on a wedged socket) while the disconnect drains.
  When the keepalive watchdog declares the peer dead and tears down, the
  reconnect path releases the lease *inside* that window. `release()` gates
  removal on `!entry.session.isConnected || ...`, so the removal branch is
  skipped: the dead transport is kept as a "warm" lease for up to 60 s, the
  lease emits `Idle` instead of `Closed(KeepaliveDead)`, and the
  `keepalive_dead` attribution is lost from the reconnect trail.
- **Failure scenario**: flaky Wi-Fi → half-open transport → keepalive death →
  async close → EOF-driven reconnect releases the poisoned lease → dead lease
  survives as warm; next attach may reuse it.
- **Confidence**: confirmed-by-trace.
- **Fix direction**: expose a close-initiated signal (`isClosing` /
  `closeStarted` already exists internally) on `SshSession` and route
  `release()` through it; or have keepalive-dead teardown proactively evict the
  lease and emit the `KeepaliveDead` edge.

#### [MEDIUM] C2 — `acquire()` / `hasLiveOrConnectingLease` can hand a *dying* transport to a concurrent same-host acquirer

- **Where**: `SshLeaseManager.kt:126-133` (reuse path), `:429-433`, `:450-455`.
- **What**: same root as C1 — every liveness predicate keys off
  `session.isConnected`. During the async-close window a concurrent same-host
  acquire (folder probe, agent-kind exec, the reconnect's own `ensureLease`)
  takes the reuse branch and gets a transport that is mid-teardown; ops throw
  `TransportClosedException`. Self-heals via the stale-lease auto-recover
  retry, so the net effect is a slower, extra-round-trip keepalive-death
  recovery and a wrong "Attaching…" affordance.
- **Confidence**: confirmed-by-trace (stale read); timing-dependent trigger.
- **Fix direction**: same primitive as C1 — exclude close-initiated sessions
  from reuse/`hasLiveLease`.

#### [LOW] C3 — `ConnectionController` reducer relies on undocumented Main-thread confinement

- **Where**: `shared/core-connection/.../ConnectionController.kt:72-91`
  (non-atomic read-modify-write of `_state.value`), `:57` (`graceDeadlineMs`
  plain `var`).
- **What**: safe today because all callers are Main-confined, but nothing
  documents or asserts it. A future off-Main `submit()` introduces silent
  lost-update races in the most critical subsystem.
- **Fix direction**: document the contract on `submit()` + add a debug-build
  dispatcher assertion (or synchronize and make `graceDeadlineMs` `@Volatile`).

**Architectural health**: one small consolidation away from clean — a single
authoritative "close initiated" liveness state. The known dual-authority
residue is already tracked as #1047 and remains the biggest lever.

### 1.2 Terminal / tmux render pipeline

Defense-in-depth is mature (frame-budgeted VT parse, coalesced render signal,
capped buffers, layered heal apparatus). No new main-thread-blocking parse, no
unbounded scrollback growth, no pipe leaks across reconnects. Remaining gaps:

#### [MEDIUM] T1 — Structural tmux events share a lossy 256-slot bus with high-rate `%output` copies

- **Where**: `shared/core-tmux/.../TmuxClient.kt:824-827` (`eventBus`,
  `extraBufferCapacity = 256`), `:2098-2120` (every `%output` copied onto the
  bus), `:2327-2350` (structural events use the same `tryEmit`). Sole consumer
  `app/.../TmuxSessionViewModel.kt:8053-8063`; the `%output` copy feeds only
  `logFirstPaneOutput`.
- **What**: `tryEmit` silently drops when the collector lags behind an output
  burst. A dropped `%output` copy is harmless; a dropped
  `%window-close`/`%session-changed`/`%layout-change` is a state desync —
  stale `[wN]` node in the tree or wrong active session until the next
  structural event happens to re-trigger reconciliation.
- **Confidence**: mechanism confirmed-by-trace; drop frequency load-dependent.
- **Fix direction**: stop multiplexing `%output` onto the structural bus (move
  the first-frame log to a tap on `outputFor()`), or give structural events
  their own never-shared channel.

#### [LOW] T2 — Control-reader LF-framing buffer is uncapped

- **Where**: `TmuxClient.kt:1844-1882` (`lines` flow), `takeLine` `:2604-2611`.
- **What**: bytes accumulate until `0x0A` with no ceiling; an LF-starved stream
  (binary MOTD, degraded non-control-mode stream) grows the buffer until OOM.
- **Fix direction**: cap the in-progress line (few hundred KB); on overflow,
  emit-and-reset with a diagnostic.

#### [LOW] T3 — Shell panes still run four independent full-viewport regex scans per frame on Main

- **Where**: `shared/core-terminal/.../TerminalSurface.kt:471-505` (URL scan),
  `:643-704` (SmartSelection + FilePath + EngineCommand).
- **What**: #796 moved *agent*-pane scanners off-main; shell panes
  (`affordanceScannersEnabled = true`) still pay ~4 redundant viewport
  extractions + regex passes per rendered frame. Frame-coalesced, so bounded —
  but a heavy shell stream (build log, `cat` of a large file) with the keyboard
  up costs sustained main-thread time on a mid-range device.
- **Fix direction**: extract the viewport text once per frame for all four
  matchers, or reuse the off-main debounced overlay agent panes already use.

#### [LOW] T4 — No escape from a never-closed `%begin` block → total silent render freeze

- **Where**: `shared/core-tmux/.../protocol/ControlEventStream.kt:57-93`.
- **What**: while `openBlock != null` every line (including real `%output`) is
  routed to command-response payload. A lost/garbled `%end` latches the block
  forever: the pane freezes although the transport is alive, and the heal
  oracles are blind because bytes simply stop reaching the model.
- **Confidence**: unlikely trigger, severe blast radius.
- **Fix direction**: bound an open block by lines/bytes/time; on exceeding,
  abandon the block, resume event parsing, record a diagnostic.

### 1.3 App layer (lifecycle, storage, services, share/update)

Notably good shape: untrusted server-output parsers uniformly degrade instead
of crash, PendingIntents are immutable, FGS notifications careful, launch path
de-blocked (#951), traced `!!` sites all safe. Residual risk is failure
containment on cold launch plus guard asymmetries.

#### [MEDIUM] A1 — Corrupt prefs file crash-loops the app at launch, no recovery

- **Where**: `app/.../settings/SettingsRepository.kt:109-132`.
- **What**: the eager `warmUpScope.async { getSharedPreferences(...); readSnapshot(...) }`
  has no try/catch; both consumers resolve it via `runBlocking { await() }` on
  Main. A corrupt/truncated `app_settings.xml` (power loss, disk-full) throws
  inside the async, is latched in the `Deferred`, and rethrows at first
  composition — an unrecoverable crash-loop, since settings underpins the theme
  at the composition root. Secondary: the `runBlocking` itself is a cold-launch
  stall risk when the preload loses the race (the #1087/#1088 class, mitigated
  not eliminated).
- **Confidence**: rethrow mechanism confirmed-by-trace; trigger plausible.
- **Fix direction**: `runCatching { ... }.getOrDefault(defaults)` (best-effort
  clear the corrupt file); consider seeding `_settings` with defaults and
  reconciling when the preload lands (weigh against the no-theme-flash goal).

#### [LOW] A2 — `ForwardingService.startForeground` unguarded (sibling service is guarded)

- **Where**: `app/.../portfwd/service/ForwardingService.kt:343-364` vs the
  guarded `sessions/service/SessionConnectionService.kt:156-174`.
- **What**: `ForegroundServiceStartNotAllowedException` (Android 12+) is
  uncaught; latent because current callers are foreground-gated, but a
  background-edge start (network-restore rebuild landing on the app→background
  transition) crashes the process.
- **Fix direction**: mirror the sibling — `runCatching`, on failure `stopSelf()`.

#### [LOW] A3 — QR share export crashes on any multi-chunk payload

- **Where**: `app/.../hosts/HostListViewModel.kt:748` —
  `QrChunkCodec.encode(importPayload).single()` in a bare `viewModelScope.launch`.
- **What**: `encode` is a multi-part API (1500-byte chunks); `.single()` throws
  for >1 chunk and the uncaught throw crashes the app. Import side handles
  multi-part fine; export cannot produce it. Trigger needs unusually long
  host/key names, but it's a hard cliff with no guard.
- **Fix direction**: render all chunks (the UI the codec was built for), or at
  minimum `runCatching` + surface `_shareMessage`.

#### [LOW] A4 — Hygiene batch: ReleaseChecker leak, ShareActivity unparcel, Room migration gap

- `app/.../release/ReleaseChecker.kt:110-144`: never `disconnect()`s, never
  drains/closes `errorStream` on non-200 (the common GitHub 403) — fd/socket
  hygiene over a long dogfood session.
- `app/.../share/ShareActivity.kt:47,166-235`: exported activity unparcels
  `EXTRA_STREAM` unguarded; a malformed share intent can crash it
  (`BadParcelableException`/CCE pre-33).
- `shared/core-storage/.../AppDatabase.kt:39,250-258` + `app/.../di/StorageModule.kt:52-56`:
  no migration edge from schema 2–7 or 9 (8→10 skips 9) and destructive
  fallback allowlists only v1 — a DB at those versions crash-loops with no
  recovery. Latent (those versions likely never shipped) — either document
  that or extend the stale-schema allowlist; enable `exportSchema` (existing
  TODO) so future gaps are caught in CI.

### 1.4 Agent-awareness / conversation / voice / usage

The JSONL parsers themselves are the sturdiest code audited (consistent
`runCatching`/skip). The risk sits in the *plumbing* that feeds them and in
shell-side string extraction.

#### [HIGH] G1 — Usage panel: one malformed/extra stdout line discards ALL providers (the #847 class)

- **Where**: `shared/core-usage/.../PocketshellUsageJsonParser.kt:74-79` (throws
  on first bad record), `app/.../usage/UsageRemoteSource.kt:169-174` (exit-0
  path wraps any throw into whole-panel `Failed`).
- **What**: a single drifted record (`short_term` not-an-object, non-numeric
  `percent_remaining`) or one non-JSON preamble line (MOTD, wrapper log line)
  discards every healthy provider. Directly contradicts
  `docs/usage-panel.md` ("older hosts still render it"). Asymmetric: the
  exit≠0 path already degrades gracefully.
- **Confidence**: confirmed-by-trace.
- **Fix direction**: per-record resilience on the success path — skip records
  that fail, return the providers that parsed.

#### [MEDIUM] G2 — Cold-open transcript read has no byte cap → OOM risk

- **Where**: `app/.../session/AgentConversationRepository.kt:1237-1240`,
  `1296-1301`, `961`.
- **What**: tail reads are bounded by *line count* only (~240 lines first
  paint); one Claude JSONL line can be multi-MB (base64 image, huge
  tool_result). Full stdout is materialised as a `String` then `.split("\n")`
  roughly doubles it — hundreds of MB heap on a pathological transcript.
- **Fix direction**: server-side byte cap on the tail (`head -c` after `tail`,
  drop truncated leading line) or per-line length clamp in the parser.

#### [MEDIUM] G3 — Inline dictation transcript silently dropped when focused pane is null / re-keys mid-transcribe

- **Where**: `app/.../tmux/TmuxSessionScreen.kt:1140-1152` (collector,
  `return@collect` on null pane) consuming
  `InlineDictation.kt:172` (`MutableSharedFlow(replay = 0)`, emit at `:641`).
- **What**: emit with no active collector is dropped; the collector is a
  `LaunchedEffect` keyed on `focusedPaneId`, which cancels/restarts on pane
  churn during the 1–3 s Whisper round-trip; null pane → discard. FSM already
  returned to Idle with no error — silent loss of a successfully transcribed
  command, exactly the loss class the pending-queue path exists to prevent.
- **Confidence**: null-pane drop confirmed-by-trace; re-key race plausible.
- **Fix direction**: small replay/buffer on the flow, route undeliverable
  transcripts through `PendingTranscriptionStore`, surface a retry affordance
  instead of `return@collect`.

#### [MEDIUM] G4 — Codex/agent-log version-skew fragility: sed cwd extraction + first-line envelope assumption

- **Where**: `AgentConversationRepository.kt:1774-1783` (Codex `detectionCommand`
  grep/sed of `session_meta` → `payload.cwd`, exact-match else `continue`);
  `:1693-1704` (`parseAgentLogEnvelopeLines` commits to the first non-blank
  line being the JSON envelope).
- **What**: a Codex rollout schema tweak (field moved/renamed, pretty-printed
  JSONL) silently zeroes all candidates → Conversation never binds; a host
  helper printing any preamble line before the envelope makes the whole window
  read as empty. Both are the "old/mismatched CLI silently breaks a surface"
  pattern (#847 class) with no diagnostic.
- **Fix direction**: tolerant cwd extraction with fd-ownership fallback (already
  implemented for the ambiguous case); scan for the first line that parses as a
  JSON object carrying `lines` instead of the first non-blank line.

#### [MEDIUM] G5 — Masked-agent fallback binds cross-kind by mtime → can bind the wrong session (the #819/#807 class)

- **Where**: `AgentConversationRepository.kt:604-632`, kind pick at `:622`
  (`candidates.maxByOrNull { it.modifiedAtMillis }?.agent`).
- **What**: the "trust a live transcript" fallback picks the agent kind of the
  most-recently-modified candidate across ALL kinds in the cwd; only Codex uses
  fd ownership. Two different-kind agents sharing a cwd → the busier sibling's
  transcript wins and the pane's Conversation binds to the wrong session/kind.
  A distinct new instance of the wrong-binding class behind #962/#1057/#1158.
- **Fix direction**: extend `/proc/<pid>/fd` ownership beyond Codex in this
  fallback; refuse to bind when >1 kind is present without an ownership signal.

#### [LOW] G6 — Voice hygiene: unbounded PCM buffer, busy-spin on read==0; adjacency dedup collapses genuine repeats

- `shared/core-voice/.../PcmCapturePump.kt:72,102-127`: no cap on capture
  duration/bytes (~19 MB / 10 min plus WAV copy + multipart body); reader loop
  spins hot if `source.read` returns 0.
- `AgentConversationRepository.kt:164-180`: the #819 duplicate-turn collapse is
  content+adjacency based, so a genuinely repeated short message ("ok" twice)
  is lossy-merged. Scope the collapse to the known echo shape.

## Part 2 — UX jobs-to-be-done audit

| # | Job | Support | Biggest friction |
|---|-----|---------|------------------|
| 1 | Connect fast & stay connected | Adequate | 3-tap tree walk every time; widget is a dead count |
| 2 | Monitor an agent / read conversation | Strong design / weak reliability | #1158 recurrence; no cross-session "who needs me" glance |
| 3 | Steer the agent with minimal typing | Strong | no one-tap approval reply for "proceed? (y/n)" |
| 4 | Juggle sessions/projects/hosts | **Weak** | cross-host session list computed but never surfaced |
| 5 | Ordinary shell commands on phone keyboard | Adequate | Esc/Ctrl-C/arrows behind a sheet; D12 regressed |
| 6 | Know provider quota/usage | Strong | buried 2–3 taps deep for the healthy-case glance |
| 7 | Recover from network drops | Strong | >5 min gap = full visible reconnect (Mosh deferred, O2) |
| 8 | Onboard a new host | Adequate | copy-only empty state; no guided key→host→test wizard |
| 9 | Review files/diffs, leave feedback | Adequate | no in-app diff — bounces to GitHub web |
| 10 | Know when an agent needs input | Adequate (new) | hooks install not part of host bootstrap → silent hosts |

Top-ranked improvements (impact-on-daily-dogfood × effort):

1. **Escalate #1158** (Conversation tab disappears) to release-blocker weight
   with a D31 class test — the product's core surface intermittently vanishes.
2. **Cross-host "Active sessions" surface** — one glance + one tap to any live
   agent on any host; `SessionsDashboardViewModel` already computes the
   aggregate. *Needs maintainer sign-off: partially reverses the #418/#171
   declutter (deliberate removal).* (M)
3. **Agent-approval quick-reply row** — one-tap `Yes`/`No`/`Enter`/`1 2 3` when
   an agent pane is waiting for input; the highest-frequency interaction in the
   supervise-agents workflow. (S–M)
4. **Fold `pocketshell hooks install` into host bootstrap** so "agent needs
   input" notifications work by default on every host (D21/D26-compliant). (M)
5. **Agent-state chip** (idle/working/waiting) on host cards + session rows,
   fed by the stop-hook bus; complements #859. (M)
6. **Minimal always-visible micro key-row** (`Esc` `Ctrl-C` `Tab` arrows);
   full grid stays in the sheet. *Touches D12 vs #784/#789 — `decisions.md`
   D12 is stale either way and must be reconciled.* (S–M)
7. **One-tap "Resume last session"** on host card + widget deep-link into a
   session (widget currently opens the host list). (M)
8. **Glanceable usage pill** on the landing app bar for the healthy case. (S)
9. **In-app unified diff view** (reuse the monospace renderer with +/- gutters),
   pairing with the existing review-comments flow. (M)
10. **Guided first-run** — primary "Add your first host" button on the empty
    state + short key→host→test-connect wizard. Low daily weight; the
    shareability gate. (M)

Locked-decision touchpoints: item 2 reverses #418/#171 (maintainer call);
item 6 sits on the D12/#784 doc-code conflict (reconcile `decisions.md`
regardless of the product call); the cross-session glance must stay a *state
indicator* to remain inside D14/D15 scope; Mosh remains deferred (O2) and is
deliberately not proposed.
